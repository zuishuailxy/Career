**MCP（模型上下文协议）是一个开放标准，旨在为AI应用与外部数据源及工具之间的交互提供一种标准化的“通用语言”。**

由Anthropic于2024年底推出，MCP被广泛视为AI领域的“USB-C接口”。它解决了过去每接入一个新数据源（如数据库、API、本地文件）都需要编写定制化代码的“碎片化”难题。

### 🏛️ 核心架构：客户端-主机-服务器

MCP遵循客户端-主机-服务器架构，通过标准化协议让不同组件协同工作：

- **MCP主机 (Host)**：发起连接的AI应用程序，如Claude Desktop、AI驱动的IDE等。它是用户与AI交互的界面，负责管理和协调整个会话。
- **MCP客户端 (Client)**：主机内部与单个MCP服务器保持1:1连接的组件。它负责在主机和服务器之间路由协议消息。
- **MCP服务器 (Server)**：通过MCP向客户端（进而向AI模型）公开特定功能（如工具、资源、提示词）的轻量级服务。服务器可以是本地进程，也可以是远程服务。

### 🧩 核心原语 (Primitives)：MCP的三类功能

MCP服务器通过以下三种核心原语向AI模型提供上下文和功能：

- **资源 (Resources)**：服务器向AI暴露的上下文和数据，类似文件系统中的“文件”。AI可以“读取”这些资源来获取信息。例如：一个数据库查询结果、一份PDF文档的内容、或一个Git仓库的代码结构。
- **工具 (Tools)**：AI模型可以直接执行的函数。这是MCP最核心的能力之一，让AI不仅能“理解”，还能“操作”。例如：一个用于发送邮件的函数、一个用于创建Jira工单的API、或一个用于执行SQL查询的接口。
- **提示词 (Prompts)**：可复用的模板化消息和工作流。它们可以指导AI如何更好地完成特定任务。例如：一个“代码审查”提示词模板，规定了AI审查代码时应遵循的步骤和风格。

### 💡 核心设计原则

MCP的设计遵循一套明确的准则，以确保其安全性、灵活性和可扩展性。

- **用户控制 (User Control)**：用户必须明确同意并理解所有数据访问和操作。主机在向服务器暴露用户数据或执行工具前，必须获得明确的用户同意。
- **关注点分离 (Separation of Concerns)**：服务器是“无状态”的，只负责提供专注、明确的功能。复杂的工作流编排和上下文聚合由主机负责，这使服务器构建简单且高度可组合。
- **安全边界 (Security Boundaries)**：服务器被设计为“隔离”的，不能读取整个对话历史或“窥视”其他服务器。主机在中间充当“防火墙”，只向每个服务器提供其完成任务所必需的最小上下文。
- **基于能力协商 (Capability Negotiation)**：客户端和服务器在连接初始化时会明确声明自己支持的功能。这确保了双方对可用的功能（如资源、工具、提示词）有清晰的理解，并保证了协议的向后兼容性和可扩展性。

### 🚀 主流应用场景

MCP正在改变AI与外部世界交互的方式，主要应用在以下场景：

- **AI驱动的开发工具**：将MCP服务器连接到代码仓库、文档和API，让AI助手（如GitHub Copilot）能理解整个项目上下文，执行代码分析、生成和重构。
- **企业级AI助理**：允许内部AI助理安全地访问公司专有文档、CRM系统、数据库等，检索信息并执行业务操作。
- **桌面与个人助理**：通过本地MCP服务器，让AI助手安全地访问本地文件、执行系统操作、管理邮件和日程。
- **多智能体协作**：MCP为不同AI智能体之间的标准化通信提供了基础，是实现可互操作、可扩展的多智能体系统的关键技术。

### 📌 关键最佳实践

在构建MCP应用时，遵循社区总结的最佳实践可以提升质量：

- **提供丰富的服务器描述**：清晰地解释服务器的用途，以及如何将工具、资源等能力映射到具体工作流中。
- **工具命名**：使用统一的命名规范，如 `camelCase`（例如 `getNpmPackageInfo`），以确保客户端能正确发现和调用。
- **工具描述**：为工具提供详细的描述，并给出别名，帮助LLM准确理解工具的用途和触发场景。
- **避免直接1:1映射API**：不要简单地将后端API的每个端点都映射为一个MCP工具。相反，应从AI的角度出发，设计更高级、更符合任务意图的工具。
- **注重性能与安全**：对代价高昂的工具调用实现缓存；遵循最小权限原则，并对工具调用进行明确的用户授权。

### 🆚 MCP在技术栈中的定位

为了更好地理解MCP，可以将其与之前讨论的Harness Engineering、Skill、Rule等概念进行对比：

- **MCP**：是一种**连接标准**，定义AI应用（客户端）如何与外部工具（服务器）通信的“语言”和“接口”。
- **Harness Engineering**：是一种**系统工程方法**，关注如何为AI构建可靠的运行环境。MCP是实现Harness中“工具集成”这一环节的标准方式。
- **Skill**：是一个**能力封装单元**。一个Skill内部可能会利用MCP来与外部工具交互，MCP是Skill与外部世界沟通的桥梁。
- **Rule**：是一种**行为约束**，定义了AI应该遵循的准则。Rule可以在Harness层面应用，也可以在MCP服务器内部实现，以限制工具的使用方式。

因此，MCP、Skill、Rule和Harness Engineering共同构成了一个完整的AI工程化体系：**Rule** 划定边界，**Skill** 封装流程，**MCP** 提供连接，而 **Harness Engineering** 则是整合这一切、让AI系统稳定运行的工程实践。

### 💎 总结

MCP通过提供标准化的交互方式，正成为AI应用连接外部世界的核心基础设施。它解决了AI能力扩展中的“最后一公里”问题，让大语言模型（LLM）能够无缝地理解、访问和操作真实世界的数据与系统。

## 如何使用MCP

要使用 MCP（模型上下文协议），你既可以作为**使用者**，在 AI 客户端中配置现成的 MCP 服务来扩展其能力；也可以作为**开发者**，自己动手构建 MCP 服务器，为 AI 应用提供自定义的工具和数据。

可以把 MCP 想象成 AI 界的“USB-C 接口”，它通过一套统一的标准，让 AI 应用能够方便地调用各种外部工具。根据 MCP 官方数据，采用这种标准可以将工具对接的耗时从数天缩短至 **5-10 分钟**。

### 👤 作为使用者：在 AI 客户端中配置 MCP 服务

如果你希望在使用 AI 客户端（如 Claude Desktop、Cherry Studio 或 Cursor）时，让它能访问本地文件、查询数据库或调用网络 API，最直接的方式就是配置现成的 MCP 服务器。

以 **Claude Desktop** 为例，主要步骤如下:

1.  打开 Claude Desktop 的 **“设置” > “扩展功能”**。
2.  在扩展功能目录中，你可以**一键安装**由 Anthropic 审核的公共 MCP 服务器，或者通过上传 `.mcpb` 文件来**安装自定义的 MCP 服务器**。
3.  安装完成后，根据界面提示完成必要的配置（如 API 密钥），该扩展功能就会自动在你的对话中生效。

对于其他 AI 客户端，配置方式也大同小异，通常都是在设置中指定 MCP 服务器的启动命令或服务地址。

### 👨‍💻 作为开发者：构建自己的 MCP 服务器

如果你需要为 AI 提供自定义的能力，可以自己开发 MCP 服务器。目前，主流的开发语言是 **Python** 和 **Node.js**。Python 开发者可以优先使用官方或社区提供的 SDK，例如 `mcp` 或 `FastMCP` 库，来简化开发过程。

#### 1. 环境准备与依赖安装

首先，创建一个新的项目目录，并设置好 Python 虚拟环境。

```bash
# 创建并进入项目目录
mkdir my-mcp-server
cd my-mcp-server

# 创建并激活虚拟环境 (以 Python 为例)
python3 -m venv .venv
source .venv/bin/activate  # Windows 系统请使用 .venv\Scripts\activate
```

然后，安装 MCP 的 Python SDK。

```bash
pip install mcp
```

#### 2. 编写一个基础的 MCP 服务器

创建一个 Python 文件（例如 `server.py`），编写一个最简单的 MCP 服务器。

这个服务器会暴露一个名为 `echo` 的工具，功能是原样返回输入的信息。

```python
# server.py
from mcp.server.fastmcp import FastMCP

# 1. 创建一个 MCP 服务器实例
mcp = FastMCP("我的第一个MCP服务器")

# 2. 使用 @mcp.tool() 装饰器暴露一个工具函数
@mcp.tool()
def echo(message: str) -> str:
    """将输入的消息原样返回。"""
    return f"Echo: {message}"

# 3. 让服务器以标准输入输出 (stdio) 模式运行
if __name__ == "__main__":
    mcp.run(transport="stdio")
```

**代码解析**：

- `@mcp.tool()` 装饰器是关键，它将 Python 函数 `echo` 暴露为一个 MCP 工具。
- 函数的**类型注解**（`: str`）和**文档字符串**（`"""..."""`）会被自动解析，用于生成工具的 `description` 和 `inputSchema`，帮助 AI 理解工具的用途和参数。
- `mcp.run(transport="stdio")` 启动服务器，并通过标准输入输出与客户端通信。

#### 3. 测试你的 MCP 服务器

你可以使用 MCP 官方提供的 **MCP Inspector** 工具来测试服务器。

首先，安装 MCP Inspector：

```bash
brew install mcp-inspector
```

然后，用 Inspector 来启动你的服务器并进行测试：

```bash
mcp-inspector python server.py
```

Inspector 会提供一个 Web 界面，让你可以查看服务器暴露的工具，并直接调用它们来验证功能。

#### 4. 编写一个 MCP 客户端（可选）

你也可以编写代码来主动连接并调用 MCP 服务器。一个最简的客户端实现包括以下步骤：

1.  通过 `stdio` 协议连接到 MCP 服务器。
2.  初始化客户端会话。
3.  列出服务器上所有可用的工具。
4.  调用特定的工具并传入参数。
5.  处理工具返回的结果。

一个完整的 Python 客户端示例可以参考这个 GitHub 仓库。

### 💎 总结与关键实践

无论是使用还是开发 MCP，遵循一些关键实践能让你的体验更好：

- **明确需求与设计**：在设计 MCP 服务器前，先明确你的 AI 应用需要什么样的工具和数据。
- **善用描述**：工具的**名称**、**描述**和**参数 Schema** 是 AI 判断是否调用它的唯一依据，务必准确清晰。
- **本地开发用 STDIO，云端部署用 SSE/HTTP**：在本地开发测试时，使用 `stdio` 模式最为简单。如果你的服务器需要部署在云端供远程 AI 应用调用，则应该使用基于 HTTP 的 **SSE (Server-Sent Events)** 或更新的 **Streamable HTTP** 传输模式。
- **保证安全**：永远不要在 MCP 服务器中硬编码 API 密钥或密码。对于生产环境的服务器，务必添加请求超时、结构化错误处理和安全的 API 身份验证机制。
