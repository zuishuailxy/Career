在AI工程化（Harness Engineering）的语境下，构建规则引擎的核心目标是**为不可预测的大模型提供一套可依赖、可解释的“确定性骨架”**。它通过将“什么能做、什么禁止”的**规则（Rule）** 从自然语言提示词中抽离出来，固化为可被代码强制执行的**硬性约束**，来确保AI的行为始终在安全、合规的边界内运行。

### 🤔 为什么需要规则引擎？

单纯依赖大模型（LLM）进行决策存在天然短板：

- **不可控与幻觉**：模型可能编造不存在的信息，或在复杂上下文中遗忘关键指令。
- **无法保证合规**：无法像代码一样100%确保遵守敏感业务规则。
- **维护成本高**：将所有规则写在提示词（Prompt）中，会导致提示词臃肿不堪，难以管理和迭代。

规则引擎的价值正是在于**为大模型这匹“野马”套上“缰绳”**，用确定性的逻辑来兜底不确定性的智能。

### 🏗️ 核心架构：如何构建一个规则引擎？

一个工业级的规则引擎通常遵循分层解耦的设计，以保证其灵活性和可维护性。

- **规则定义层 (Rule Definition)**：这是规则的“仓库”。推荐使用声明式的、人类可读的格式（如YAML或JSON）来定义规则，而不是硬编码在程序里。这便于非技术人员理解和修改。

  ```yaml
  # 规则示例 (YAML格式)
  - id: "SQL_Injection_Prevention"
    description: "防止SQL注入漏洞"
    severity: "critical"
    condition: "sql_query contains '$'"
    action: "block_and_alert"
    suggestion: "使用参数化查询替代字符串拼接"
  ```

- **规则解析与执行引擎 (Engine)**：负责加载、解析规则文件，并根据输入的上下文（Context）进行条件匹配，然后触发对应的动作（Action）。
  - **高效匹配**：对于复杂规则，可以引入 **RETE 算法**。它通过预先编译一个“判断网络”，能将匹配复杂度从 O(M×N) 降至接近 O(N)，从而支撑高并发场景。
- **上下文感知层 (Context Awareness)**：这是实现精细化控制的关键。引擎不应只做简单的条件判断，而应能感知丰富的**上下文（Context）**，如用户画像、会话状态、环境参数等。

### 🧠 与大模型的协作模式：“骨架与大脑”

在AI应用中，规则引擎与大模型并非替代关系，而是典型的**协同关系**。

- **规则引擎 = 确定性骨架**：负责处理**已知的、明确的、需要严格遵守**的逻辑。例如：
  - **安全红线**：直接拦截包含敏感词或SQL注入风险的输入。
  - **合规校验**：检查交易金额是否超限，是否需要人工审核。
  - **流程控制**：强制要求AI在完成任务后必须运行编译和测试。
- **大模型 = 柔性大脑**：负责处理**开放的、模糊的、需要理解和生成**的任务。例如：
  - 理解用户用自然语言提出的复杂问题。
  - 根据规则引擎的校验结果，生成人性化的解释和回复。

这种架构的典型工作流是：**用户输入 → 规则引擎（硬性过滤与校验）→ 大模型（理解与生成）→ 规则引擎（输出合规性检查）**。

### 💻 技术选型与落地实践

你可以根据自己的技术栈和项目复杂度，选择合适的实现路径。

1.  **轻量级实现 (适合Python项目)**
    - 如果逻辑不复杂，可以自己实现一个简单的规则匹配器。
    - 也可以使用轻量级库，如 `rule-engine`、`simple-rule-engine`或 `easy-rule-engine`。

2.  **企业级框架 (适合Java/企业级项目)**
    - **Drools**：Java生态中最成熟的规则引擎之一，功能强大，支持复杂的规则编排。
    - **LiteFlow**：轻量级的组件式流程引擎，支持规则的热加载。

3.  **其他语言的选项**
    - **Go语言**：`RuleGo` 是一个高性能的组件化规则引擎。
    - **.NET**：`NRules` 是一个基于RETE算法的开源规则引擎。

4.  **在Harness Engineering框架中的实践**
    - 可以将规则引擎作为 `HarnessController` 的一个核心组件来构建。
    - **`jsharness`** 等工具，提供了一套通过 `Rules`（规则）+ `Skills`（技能）+ `Gate`（门禁）来约束AI编码的工程化方案。

### 💎 总结

构建规则引擎的最佳实践，本质上是**将AI应用开发的确定性部分与智能部分进行分离**。一个优秀的规则引擎实践，通常具备以下特征：

- **声明式配置**：规则与代码解耦，存储在独立的、可读性强的文件中。
- **版本控制**：像管理代码一样，用Git管理规则文件的变更。
- **分层设计**：将规则按业务领域或功能类型进行清晰分组。
- **上下文感知**：规则引擎能感知丰富的上下文信息，实现精准触发。
- **明确依赖**：规则之间的依赖关系是显式声明和管理的。
- **硬性门禁**：最终通过脚本（Script）或门禁（Gate）将规则落地为“不通过就无法继续”的硬性检查。

通过构建这样一个“规则骨架”，你才能将大模型的非凡创造力，安全、可靠地固定在业务所需的轨道上。

# jsharness

`jsharness` 是一个 **Harness Engineering 的工程化工具**，能帮你把分散在文档里的团队规范，自动注入到 AI 编程助手的配置中。它的核心价值在于，让 AI 助手（如 Cursor、Copilot）在编码时能自动遵守团队的编码规范、技术栈选择和安全红线。

### 🎯 核心机制：三层约束

`jsharness` 基于 Harness Engineering 的核心理念，通过三层机制来约束 AI 的行为：

1.  **Rules (软约束)**：Markdown 格式的声明式规范，告诉 AI “什么不能做”。
2.  **Skills (半硬约束)**：步骤化的操作手册，指导 AI “具体怎么做”。
3.  **Gate Scripts (硬约束)**：可执行的检查脚本，作为最后的把关者，“不通过就拦截”。

### 🚀 快速开始：30秒初始化

在你的项目根目录下执行以下命令即可：

```bash
npx jsharness init
```

这条命令会自动完成检测、扫描、转换和写入，让规范在你下一次 AI 对话中立即生效。你也可以通过 `--tool` 参数指定目标工具，或通过 `--stack` 参数只注入特定技术栈的规则。

### ⚙️ 工作原理

1.  **扫描**：扫描项目中的 `.harness/rules/*.md` 和 `.harness/skills/*.md` 文件。
2.  **检测**：自动检测当前项目使用的 AI 工具。
3.  **转换**：将 Markdown 格式的规则转换为目标工具的原生配置格式。
4.  **写入**：将转换后的配置写入到对应工具的配置位置。

### 📁 系统架构与内置内容

`jsharness` 在项目中创建一个 `.harness/` 文件夹，内置了丰富的规则和技能模板。

- **规则层 (Rules)**：包含通用编码规范、Commit 信息格式规范、安全红线（XSS/CSRF/注入），以及 Vue3 前端技术栈规则、Java Spring Boot 后端规则、Web 应用通用规则等。
- **技能层 (Skills)**：提供步骤化的操作手册，指导 AI 执行特定任务。

### 🛠️ 支持的 AI 工具

`jsharness` 支持将规则注入到多种主流 AI 编程工具中：

| AI 工具            | 注入位置                                   | 规则支持 | 技能支持 |
| :----------------- | :----------------------------------------- | :------- | :------- |
| **CodeBuddy**      | `.codebuddy/rules/` + `.codebuddy/skills/` | ✅       | ✅       |
| **Cursor**         | `.cursorrules` (合并为单文件)              | ✅       | —        |
| **GitHub Copilot** | `.github/copilot-instructions.md`          | ✅       | —        |
| **Windsurf**       | `.windsurfrules` (XML 格式)                | ✅       | —        |
| **Continue**       | `continue.config.json` (JSON 格式)         | ✅       | —        |
| **Cline**          | `.clinerules`                              | ✅       | —        |

### 💡 与其他工具的区别

- **与 JT Harness 的区别**：`jsharness` 是面向 AI 编码助手的**行为约束与规范注入工具**，而 JT Harness 是 OpenJDK 的**通用 Java 单元测试框架**。两者在目标、应用场景和实现方式上完全不同。
- **与 Sharness 的区别**：`jsharness` 是 AI 工程化工具，而 Sharness 是基于 Shell 脚本的**自动化测试框架**。

### 💎 总结

简单来说，`jsharness` 提供了一种**标准化、自动化的方式**，来管理和实施 AI 编码规范。它将散落在文档、口头约定中的团队最佳实践，固化为 AI 可直接遵循的、机器可读的规则，让 AI 从“随性发挥”变得“有章可循”。
