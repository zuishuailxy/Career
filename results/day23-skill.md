在大模型的语境下，“Skill”（技能）是一种**模块化、可复用的能力封装单元**。它旨在将大模型从“能聊天”的通用工具，升级为“会做事”的专业助手。

你可以把它理解为给大模型配备的 **“岗位说明书”和“操作手册”** 。当模型需要完成特定任务时，它会调用相应的 Skill，按照手册中的指引去执行。

### 🆚 核心区别：Skill vs. 其他概念

Skill 在整个大模型技术栈中有着清晰的定位，它与其他概念协同工作，共同构建复杂的AI应用。

| 概念                     | 核心定义                                         | 打个比方                                  | 关系 |
| :----------------------- | :----------------------------------------------- | :---------------------------------------- | :--- |
| **大模型 (LLM)**         | 提供通用智能和推理能力的“大脑”                   | 一位智商超群、知识渊博的**实习生**        | 基础 |
| **Skill (技能)**         | 封装了特定任务知识、流程和代码的 **“能力包”**    | 发给实习生的 **“岗位说明书”和“操作手册”** | 增强 |
| **Agent (智能体)**       | 能自主规划、决策并完成复杂任务的**完整系统**     | 能独当一面的**正式员工**                  | 承载 |
| **MCP (模型上下文协议)** | 标准化 AI 与外部工具、数据源连接的 **“USB接口”** | 电脑上的**通用接口**，用于连接各种外设    | 连接 |
| **Function Calling**     | 让模型能**调用外部函数**的机制                   | 一个基础的**开关或按钮**                  | 动作 |

---

### 🧩 Skill 的构成与运行机制

一个标准的 Skill 通常包含以下部分：

- **元数据 (Metadata)**：如技能名称、描述等，方便模型检索和调用。
- **指令 (Instructions)**：核心部分，即详细的提示词或工作流程（SOP）。
- **资源 (Resources)**：可选的脚本、模板、示例等辅助文件。

所有文件通常打包在一个文件夹中，并以一个 `SKILL.md` 文件作为入口。

它的工作流程可以概括为“渐进式披露”：

1.  **发现**：模型在启动时，只会加载所有 Skill 的轻量级元数据（名称和描述），占用极少 token。
2.  **选择**：当需要执行特定任务时，模型会根据元数据判断并选择最合适的 Skill。
3.  **加载**：模型会读取该 Skill 的完整指令和相关资源，获取完成任务所需的所有知识。

### 🛠️ 主流平台与实现

各大 AI 厂商和社区都有自己的 Skill 实现方案：

- **OpenAI Skills**：将其定义为“可复用、可共享的工作流”，是一个包含指令、脚本和资源的文件夹，可通过 API 上传和管理。
- **Claude Skills**：由 Anthropic 于 2025年10月推出。作为一个能力扩展包，可自动调用。核心优势在于能封装企业级的工作流程和标准。
- **Agent Skills**：社区中更通用的一个概念。它像一个“入职培训指南”，能封装领域知识、业务逻辑和执行脚本，让智能体按标准作业程序（SOP）稳定交付任务。

### 🎯 Skill 与 Agent 的关系

要准确理解 Skill，关键在于厘清它与 Agent 的关系。**Skill 是 Agent 的“能力组件”，但 Agent 本身是一个“完整的任务执行主体”**。

大模型加上一个或多个 Skills，可以成为一个更稳定的**任务助手**，但要成为一个能**自主完成复杂任务的 Agent**，还需要具备目标感、规划能力、状态管理和执行闭环等要素。

### 💎 总结

总的来说，**Skill 是大模型生态中，将“通用智能”转化为“专业能力”的关键桥梁**。它通过封装特定领域的知识和流程，让大模型能够稳定、高效地处理各种专业化任务，是从“对话工具”迈向“自主智能体”的核心一步。

## 如何创建skill

创建一个 Skill 的核心，是**将你希望 AI 执行的特定工作流程和知识，打包成一个结构化的、可复用的“能力包”**。

最通用的方法是手动创建一个包含 `SKILL.md` 文件的文件夹。`SKILL.md` 是这个技能包的“说明书”和“入口文件”。

下面我以 **Claude** 和 **ChatGPT (OpenAI)** 为例，介绍具体的创建方法。

### 🛠️ 方法一：通过自然语言对话创建（最简单）

这是最便捷的方法，你只需用自然语言描述你想要的技能，AI 会帮你完成所有技术细节。

- **在 Claude 中**：开启一个新对话，直接说“**我想创建一个 Skill，请引导我完成**”。Claude 会启动一个引导流程，询问技能名称、功能、预期用途等细节，并最终生成一个包含所有文件的 ZIP 包供你下载。
- **在 ChatGPT 中**：同样，你可以直接告诉它“**帮我创建一个技能 (Build me a skill)**”。ChatGPT 会通过提问来引导你完成整个创建过程。

### 📁 方法二：手动创建 Skill 文件夹（更灵活）

如果你希望对技能有更精细的控制，可以手动创建。无论是 Claude、ChatGPT 还是其他支持 Agent Skills 开放标准的平台，核心步骤都是一致的。

1.  **新建项目文件夹**：在你的电脑上创建一个文件夹，命名应清晰反映技能用途，建议使用连字符（-）连接，例如 `my-first-skill`。

2.  **创建核心文件 `SKILL.md`**：在刚创建的文件夹里，新建一个文件，**必须**命名为 `SKILL.md`（注意大小写）。

3.  **编写 `SKILL.md` 内容**：使用任何文本编辑器打开 `SKILL.md`，并按以下结构编写内容。具体规范可以参考 [Agent Skills](https://agentskills.io/home) 开放标准。
    - **Frontmatter (元数据)**：文件开头必须包含 YAML 格式的元数据。其中 `name` 和 `description` 是**必需的**，AI 主要依据 `description` 来判断何时使用该技能。

      ```yaml
      ---
      name: 你的技能名称
      description: 清晰描述技能的功能和触发场景。这是AI判断是否调用的关键。
      ---
      ```

      你还可以添加 `dependencies` 等可选元数据来声明依赖的软件包。

    - **Markdown 正文 (指令主体)**：Frontmatter 之后，就可以用 Markdown 格式来编写具体的操作指令了。这部分是技能的核心，应清晰地告诉 AI：
      - 这个技能的**目标**是什么。
      - **何时**应该被触发。
      - 具体的**执行步骤**和**工作流程**。
      - 输入和输出的**格式要求**。
      - **约束和边界**，即什么能做、什么不能做。

4.  **添加辅助文件（可选）**：为了让技能更强大，你可以在文件夹中添加其他资源。例如：
    - `scripts/` 文件夹：存放 Python、Bash 等可执行脚本。
    - `references/` 文件夹：存放更详细的参考文档。
    - `assets/` 文件夹：存放模板、图片等资源文件。

### ✍️ 一个完整的 `SKILL.md` 示例

下面是一个“小红书文案生成器”的示例，你可以参考它的结构和写法。

```yaml
---
name: Rednote Copywriter
description: 专用于生成小红书风格的种草文案。当用户提到“写小红书”、“种草”、“红书文案”时激活。
---

# 角色设定
你现在是一名资深的“小红书”爆款文案专家。你深谙平台规则，擅长使用Emoji、分段、以及夸张的口语化表达。

# 核心规则
输出格式：必须包含三部分
- 标题：用 [火R] 开头，必须包含2-3个Emoji，字数控制在20字以内，制造悬念。
- 正文：每段不超过60字，段与段之间必须空一行，每段开头不要缩进。
- 标签：最后一行用 # 开头，生成5个相关标签。

语气要求：
- 多用“姐妹们”、“救命”、“谁懂啊”这类口语。
- 多用数字概括，比如“3个方法”、“2分钟搞定”。
- 严禁使用“首先、其次、再次”这种结构化词汇，要用“绝绝子”、“直接封神”来代替。

禁区：
- 不要出现“在当今社会”、“随着科技的发展”这种官腔。
- 不要评价自己的文案（不要说“以上是我的建议”）。

# 工作流程
当用户提供产品名或主题时：
1.  首先，理解产品的核心卖点和目标用户。
2.  然后，根据核心规则撰写标题、正文和标签。
3.  最后，直接输出完整的文案，不要附加任何额外说明。
```

### 💡 创建 Skill 的最佳实践

- **遵循渐进式披露**：保持 `SKILL.md` 文件简洁（建议不超过 500 行），将详细参考资料放在 `references/` 等子文件夹中。这能帮助 AI 高效地按需加载信息。
- **专注单一职责**：一个 Skill 应只专注于解决一个特定的、可重复的任务。
- **包含清晰示例**：在指令中提供具体的输入/输出示例，能帮助 AI 更好地理解你的意图。
- **测试和迭代**：创建完成后，务必进行测试。你可以在对话中尝试触发该技能，观察 AI 的思考过程，确认它是否正确加载并按预期工作。

### 🚀 如何安装和使用

1.  **上传/安装**：创建好包含 `SKILL.md` 的文件夹后，你通常需要将其打包（如 ZIP 文件）或在平台中指定该文件夹路径。然后通过各平台的上传功能进行安装。例如，在 Claude 中，路径通常是 `Settings → Capabilities → Add skills`。
2.  **启用**：上传后，你可能需要在设置中手动启用该技能。
3.  **触发使用**：在对话中提出相关需求，AI 应该会自动识别并调用相应的 Skill。

# 最佳实践

Skill 的最佳工业实践，核心在于将 AI 的“临时推理”转化为“可复用的本能反应”，其关键在于构建一个**模块化、可发现、按需加载**的能力体系。

### 📝 核心理念：渐进式披露 (Progressive Disclosure)

这是 Skill 体系最核心的设计，旨在解决传统 Prompt 工程导致上下文臃肿、注意力分散的问题。

- **发现问题**：传统做法将所有规则和知识一次性塞入提示词，随着项目复杂化，上下文窗口被撑满，模型注意力被稀释，且知识和项目深度耦合，几乎无法复用。
- **核心解法**：**只在需要时才加载需要的知识**。它将 Skill 的加载分为三个阶段，如同一个三层漏斗：
  1.  **发现 (Discovery)**：会话启动时，仅加载每个 Skill 的 **`name`** 和 **`description`**（元数据），这部分信息常驻上下文，用于意图匹配。
  2.  **激活 (Activation)**：当任务匹配上 `description` 时，再读取完整的 **`SKILL.md`** 文件，获取核心执行指令。
  3.  **执行 (Execution)**：在真正执行时，才按需加载 `scripts/`、`templates/` 等配套资源。

这种架构能将上下文 Token 消耗降低 **60%-80%**，并实现上下文成本的“按需投放”。

### 🧱 工程化标准：目录与文件结构

一个工业级的 Skill 遵循标准化的文件结构，以 `SKILL.md` 为核心。

- **`SKILL.md`**：必需。这是 Skill 的“说明书”和入口，包含元数据（`name` 和 `description`）和执行指令。
  - **`name`**: 技能名称（最多64字符）。
  - **`description`**: **最关键的部分**。清晰描述技能的功能和触发场景，AI 主要依据此判断是否调用（最多200字符）。
  - **`dependencies`**: 可选。声明所需的软件包，如 `python>=3.8, pandas>=1.5.0`。
- **`scripts/`**：可选。存放可执行脚本，用于处理复杂逻辑。
- **`templates/` 或 `assets/`**：可选。存放模板文件或静态资源。
- **`references/`**：可选。存放更详细的参考文档。
- **`agents/`**：可选。存放子智能体配置。

以下是一个 `SKILL.md` 的完整示例：

```markdown
---
name: Brand Guidelines
description: Apply Acme Corp brand guidelines to all presentations and documents, including official colors, fonts, and logo usage.
---

# Overview

This skill provides Acme Corp's official brand guidelines for creating consistent, professional materials.

## Brand Colors

- Primary: #FF6B35 (Coral)
- Secondary: #004E89 (Navy Blue)

## Typography

Headers: Montserrat Bold
Body text: Open Sans Regular

## When to Apply

Apply these guidelines when creating PowerPoint presentations, Word documents, or marketing materials.
```

### 💎 五大设计原则

根据工业级标准，一个好的 Skill 应遵循以下原则：

- **单一职责 (Focused)**：每个 Skill 只解决一个特定的、可重复的任务。多个专注的 Skill 比一个大型 Skill 更容易组合。
- **触发准确 (Accurate Triggering)**：`description` 必须精确，确保 Skill 在正确的时机被激活。
- **权限收敛 (Converged Permissions)**：只授予完成任务所必需的最小权限。
- **模型匹配 (Model Matching)**：任务复杂度应与所选模型的能力匹配。
- **可评测与可迭代 (Evaluable & Iterable)**：Skill 应能被测试验证，并能持续改进。

### 🚀 实施路线图：从验证到工程化

开发 Skill 是一个迭代过程：

1.  **从简单开始**：先用 Markdown 编写核心指令，再根据需要添加脚本。
2.  **明确需求**：将具体需求转化为可复用的 Prompt 或工作流。
3.  **测试验证**：在支持 Skill 的 Agent 环境（如 Claude Code、Codex）中运行并观察。
4.  **迭代优化**：根据测试结果不断调整指令和流程。

### 🆚 Skill 与 Rule、MCP 的关系

在一个完整的 Harness Engineering 体系中，它们分工明确：

- **Rule（规则）**：是“**软约束**”，定义项目的“工程准则”，明确什么能做、什么禁止。它为 Skill 划定边界。
- **Skill（技能）**：是“**操作手册**”，封装特定任务的完整工作流程。它告诉 AI “具体怎么做”。
- **MCP（模型上下文协议）**：是“**连接器**”，用于统一调用外部工具。Skill 可以利用 MCP 来执行具体操作。

### 💎 总结：走向“技能工程”

总而言之，Skill 的最佳工业实践标志着一个从“**提示词工程 (Prompt Engineering)**”到“**技能工程 (Skill Engineering)**”的范式转移。它的核心价值在于将团队的最佳实践、领域知识和工作流程，封装为**可发现、可加载、可复用**的标准化“能力包”，从而让 AI 的执行更标准、稳定和可组合。
