**JSON-RPC是一种轻量级的远程过程调用（RPC）协议**，它使用JSON（JavaScript Object Notation）作为数据格式，允许客户端调用服务器上的方法，就像调用本地函数一样。其核心设计目标就是**简单**和**轻量**。

### 🔧 核心工作原理

JSON-RPC的工作模式是典型的**请求-响应**模型，核心是**无状态**和**与传输层无关**的。

客户端会构造一个JSON请求对象并发给服务器，其中包含以下关键字段：

- **`jsonrpc`**: 协议版本，必须是 `"2.0"`。
- **`method`**: 要调用的远程方法名。
- **`params`**: 可选，调用方法所需的参数。
- **`id`**: 客户端生成的请求标识符，用于将响应与请求匹配。

服务器收到后，执行对应方法，并将结果封装在JSON响应对象中返回。如果出错，则会返回一个包含错误码和信息的错误对象。

此外，JSON-RPC 2.0支持**批量调用**（在单个请求中包含多个调用）和**通知**（不期望响应的请求）。

### 🆚 JSON-RPC vs. REST

这是大家最常问到的问题，它们的主要区别如下：

| 特性         | JSON-RPC                                   | REST                                 |
| :----------- | :----------------------------------------- | :----------------------------------- |
| **核心概念** | **动作 (Action)** 或**函数 (Function)**    | **资源 (Resource)**                  |
| **设计哲学** | “我要执行一个操作” (`method: "getUser"`)   | “我要操作一个资源” (`GET /user`)     |
| **接口风格** | 命令式，专注于“做什么”                     | 声明式，专注于“是什么”               |
| **传输协议** | **不依赖**，可运行于HTTP, WebSocket, TCP等 | **依赖** HTTP 及其方法 (GET, POST等) |
| **适用场景** | 简单的内部服务调用、API动作较多            | 面向公众的复杂API、需要缓存等Web特性 |

### 🤖 为什么 JSON-RPC 在 AI 领域如此重要？

JSON-RPC已成为当前AI Agent通信的**基础协议**，扮演着“通用语言”的角色。

- **模型上下文协议 (MCP)**：MCP作为连接AI Agent与外部工具、数据源的标准，其消息格式正是基于JSON-RPC 2.0定义的。
- **Agent2Agent (A2A) 协议**：这是一个支持AI智能体间结构化通信的开放标准，它也**基于HTTP和JSON-RPC 2.0构建**。
- **智能体客户端协议 (ACP)**：这是一个**基于JSON-RPC 2.0**的开放协议，它利用JSON-RPC的特性，实现了编辑器与AI编码Agent之间的**双向通信**。它用JSON-RPC的通知（Notifications）让Agent能向UI**实时推送更新**，并用双向请求让Agent能向编辑器请求权限（如工具调用）。

可以说，如果你在开发AI Agent应用，那么JSON-RPC就是你几乎必然会遇到的底层技术。

### 💎 总结

总而言之，JSON-RPC是一个**简单、轻量、传输无关的远程过程调用协议**。它通过标准化的JSON消息格式，让不同服务能够方便地互相调用函数。因其简洁性和跨语言特性，它已成为构建现代AI Agent间、以及Agent与工具/客户端间通信的**核心基础协议**。

## 例子

下面是一个 **JSON-RPC 2.0 在 AI Agent 场景中的真实应用示例**，演示了 Agent 如何通过 JSON-RPC 调用一个“天气查询工具”。

### 🌤️ 场景：Agent 调用天气工具 (MCP 协议)

假设你正在开发一个 **MCP (模型上下文协议) 服务器**，它暴露了一个工具 `get_weather`，Agent 可以通过 JSON-RPC 消息来调用它。

#### 1. Agent → Server: 发送请求

Agent 构建一个 JSON-RPC 请求，指定要调用的方法 `get_weather`，并传入城市名称参数。

```json
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": {
      "city": "上海"
    }
  },
  "id": 2
}
```

- **`method`**: 这里是 `tools/call`，表示这是一个工具调用请求（MCP 规范中定义）。
- **`params`**: 包含工具名称 `get_weather` 和参数 `{ city: "上海" }`。
- **`id`**: 请求标识符，用于匹配响应。

#### 2. Server → Agent: 返回结果

天气服务返回结果，服务器将其封装成 JSON-RPC 响应。

```json
{
  "jsonrpc": "2.0",
  "result": {
    "content": [
      {
        "type": "text",
        "text": "上海当前温度：22°C，晴朗。"
      }
    ]
  },
  "id": 2
}
```

- **`result`**: 包含工具执行结果，这里是一个文本内容。
- **`id`**: 与请求一致，Agent 据此知道这是对哪个请求的响应。

#### 3. 如果出错：错误响应

如果工具不存在或参数错误，服务器返回错误对象。

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32601,
    "message": "Tool 'get_weather' not found"
  },
  "id": 2
}
```

### 🚀 在实际系统中如何传输？

JSON-RPC 不关心底层传输协议，你可以通过多种方式发送这些 JSON 消息：

- **通过 HTTP POST**：将 JSON 请求放在 HTTP 体里，响应也以 JSON 返回。
- **通过 WebSocket**：在持久连接中发送 JSON 字符串，适合双向流式通信。
- **通过标准输入/输出 (stdio)**：如 Claude Desktop 与本地 MCP 服务器通信时，就通过启动子进程的 stdin/stdout 交换 JSON-RPC 消息。

### 💡 为什么这对 Agent 开发很重要？

- **标准化工具调用**：无论使用什么语言开发工具，只要遵循 JSON-RPC 格式，Agent 就能统一调用。
- **解耦**：Agent 和工具不需要共享代码库，仅依赖协议规范，便于扩展和部署。
- **可观测性**：每个请求/响应都是纯 JSON，易于记录、回放和调试。
