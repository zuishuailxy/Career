在 Harness Engineering 中，**Hooks 是一种“机械强制”的自动化拦截机制**。它通过在 AI 执行动作的关键节点上设置“关卡”，实现不依赖 AI 主观意愿的自动强制执行。

如果说 `Rules`（规则）是给 AI 的“建议”，那 **Hooks 就是撤掉它做错事的“权限”**。这种设计在工程实践中被证明非常有效：常规指令的遵从率可能只有 **20%**，而 Hooks 的机械约束能达到 **100%** 的执行率。

### ⚙️ 核心价值：命令与约束的本质区别

| 特性 | Rules / Prompts | **Hooks** |
| :--- | :--- | :--- |
| **性质** | “建议”或“指导” | “机械约束”或“权限” |
| **依赖** | AI 自主“记得”并遵守 | **系统级强制执行**，AI 无法跳过 |
| **执行率** | 低（~20%） | **高（100%）** |
| **作用方式** | 影响模型思考和输出 | 在动作执行前后**注入真实命令**并拦截 |
| **典型场景** | 规定编码规范、项目约定 | 强制代码格式化、拦截危险命令、禁止修改关键文件 |

### 🕹️ 工作原理：生命周期拦截与决策

Hooks 像“安检员”，挂在 AI 的每个动作节点上。其工作流程如下：
1.  **事件触发**：当 AI 准备调用工具、编辑文件或执行命令时，系统会产生一个事件。
2.  **上下文打包**：系统将当前操作（如修改的文件路径、执行的 shell 命令）打包成 JSON 数据，通过标准输入（stdin）传给 Hook 脚本。
3.  **脚本决策**：脚本读取 JSON 并执行判断逻辑。
4.  **返回结果**：脚本通过退出码（exit code）决定后续行为：
    *   `exit 0`：**放行**。
    *   `exit 2`：**阻断**，同时将错误信息发给 AI，让它调整行为。

### 🧩 Hooks 的类型与应用场景

根据执行时机，Hooks 可分为两大类：
*   **Pre-Hooks（前置钩子）**：在动作执行**前**触发，用于**预防和拦截**。例如：拦截 `rm -rf` 等危险 shell 命令；阻止编辑 `.env` 等受保护文件；提交前强制运行 lint 检查。
*   **Post-Hooks（后置钩子）**：在动作执行**后**触发，用于**检查和验证**。例如：文件编辑后自动格式化；记录每一步操作日志。

在实际应用中，Hooks 可以覆盖 AI 操作的整个生命周期：
*   **`SessionStart`**: 会话开始时加载项目状态。
*   **`PreToolUse`**: 工具调用前进行安全拦截和门控审查。
*   **`PostToolUse`**: 工具调用后检测循环、路由上下文。
*   **`Stop`**: Agent 停止前验证工作成果。
*   **`SessionEnd`**: 会话结束时反思并持久化进度。

### 🧰 最佳实践：将规则转化为强制约束

1.  **用 Hooks 做强约束**：对于安全红线（如禁止删除生产数据库），必须用 Hooks 实现，不能只写在 Rule 里。
2.  **与 Rules、Skills 配合使用**：例如，用 **Rule** 规定“支付模块变更必须测试先行”；用 **Skill** 定义“支付模块 TDD”的具体流程；用 **Hook** 在提交前**强制运行支付测试**作为硬性验证。
3.  **在沙箱中安全执行**：Hooks 的执行环境（沙箱）应隔离，防止 Hook 脚本本身的错误影响主系统。
4.  **让“每一个 Agent 错误都变成永久约束”**：采用“棘轮规则”（Ratchet rule），将 AI 犯过的错误固化为新的 Hook，防止问题再现。

总的来说，在 Harness Engineering 的语境下，**Hooks 是将 AI 的“不可靠行为”转化为“确定性流程”的关键技术**。它用代码和系统机制来约束 AI，从根本上解决了单纯依赖提示词（Prompt）带来的不确定性。
