**Context Engineering（上下文工程）** 是从提示工程（Prompt Engineering）演进而来的一门新兴学科。它的核心目标是**设计并构建动态系统，在恰当的时机，以恰当的格式，向大语言模型（LLM）提供恰当的信息和工具**。

如果说提示工程关注的是“如何提问”，那么上下文工程关注的则是“模型在回答问题时，应该处于一个怎样的信息环境中”。

### 📖 核心定义：不止是“提示”

要理解上下文工程，首先需要拓宽对“上下文”的定义。它不再仅仅是发给模型的单一提示词，而是模型在生成响应前所“看到”的全部内容。这包括：

- **系统提示**：定义模型角色和行为的初始指令集。
- **用户提示**：用户当前的任务或问题。
- **短期记忆**：当前的对话历史。
- **长期记忆**：跨会话的用户偏好和知识。
- **检索信息**：通过RAG等技术从外部知识库检索的相关信息。
- **可用工具**：模型可以调用的外部函数或API的定义。
- **结构化输出**：对模型响应格式（如JSON）的要求。

### 🆚 上下文工程 vs. 提示工程

两者的核心区别在于视角的转变。提示工程是一次性的“文本优化”，而上下文工程是系统性的“信息环境设计”。

| 维度       | 提示工程 (Prompt Engineering) | 上下文工程 (Context Engineering)         |
| :--------- | :---------------------------- | :--------------------------------------- |
| **关注点** | 如何写出完美的指令            | 模型工作时处于什么信息环境               |
| **对象**   | 一个静态的文本字符串          | 一个动态生成的信息系统                   |
| **方法**   | 优化措辞、添加示例            | 管理记忆、检索信息、调用工具、压缩上下文 |
| **目标**   | 让模型理解当前任务            | 让模型拥有完成任务所需的一切             |

### 🏗️ 核心技术支柱

根据一篇涵盖了超过1400篇研究论文的系统性综述，上下文工程可以分解为三个核心部分：

1.  **上下文检索与生成 (Context Retrieval and Generation)**：如何获取和生成有价值的上下文。这包括基于提示词的生成，以及通过RAG等方式从外部知识库（如文档、数据库）获取信息。
2.  **上下文处理 (Context Processing)**：在将信息送入模型前，如何对其进行处理。技术包括处理长序列、信息自我精炼（如摘要）、以及整合结构化信息。对于超长文本，需要进行有状态的压缩和摘要。
3.  **上下文管理 (Context Management)**：如何长期、高效地维护和利用上下文。这涉及构建记忆层次结构（短期/长期记忆）、上下文压缩和优化策略。

### 💡 关键实践与技巧

在实际应用中，上下文工程体现在一系列具体实践上：

- **动态构建，而非静态模板**：上下文是根据当前任务即时生成的。例如，查询日历和查询邮件的上下文是完全不同的。
- **选对信息，而非全量灌入**：通过RAG等技术，精准检索最相关的文档片段，而不是将所有文档都塞进上下文窗口。
- **格式至关重要**：信息的呈现方式直接影响模型的理解。一份简洁的摘要远比原始数据罗列有效；清晰的工具接口定义也比模糊的指令更能让模型正确调用。
- **管理“上下文衰减”**：随着上下文窗口中的token增多，模型准确回忆早期信息的能力会下降。因此，上下文必须被视为一种“有限资源”。
- **善用工具**：当模型仅凭输入无法完成任务时，应为其提供合适的工具（如搜索引擎、计算器、数据库查询接口）。
- **投资数据管道**：上下文工程很大一部分工作在于构建和维护用于注入上下文的数据管道。

### 🎯 为何如此重要？

上下文工程被视为构建可靠AI智能体（Agent）成败的关键。随着AI从单次对话（如ChatGPT）演进到能进行多步推理、工具调用的智能体，上下文的复杂度呈指数级增长。

- **决定成败**：大多数AI智能体失败，并非模型能力不足，而是因为上下文质量不佳。
- **系统级思维**：它要求工程师具备系统架构思维，而不仅仅是写提示词的技巧。
- **弥合认知鸿沟**：上下文工程本质上是将高熵的人类意图，预处理为机器可理解的低熵表示，从而弥合人机之间的认知鸿沟。

### 💎 总结

上下文工程是AI应用开发的一次范式转移，它标志着我们与AI交互的方式，从“编写更好的指令”演进到了“设计更优的信息环境”。对于希望构建可靠、高性能AI应用（尤其是复杂的AI智能体）的工程师来说，掌握上下文工程的思维和技术，正成为一项核心能力。

# 最佳工业实践

上下文工程（Context Engineering）的最佳工业实践，核心在于将AI应用的开发从依赖“写好提示词”（Prompt Engineering）的“手工艺”，升级为通过“设计信息环境”（Context Engineering）的系统工程。它不再追求一次性完美的提示，而是致力于构建一个动态、结构化的上下文系统，让AI模型在运行时总能“看到”最恰当的信息。

Gartner预测，到2028年，80%的AI应用构建工具将内置上下文工程能力。这项能力的核心目标，就是**精准控制模型的“注意力”**。如果上下文工程做得好，小模型也能有稳定的表现；做不好，即便用最好的模型，也难以避免幻觉。

### 🧠 上下文工程的四大核心策略

要构建生产级的上下文系统，通常围绕以下四个核心策略展开：

- **写入策略 (Writing)**：创建清晰、有效的上下文内容。关键在于编写模块化、可复用的结构化指令，并将提示词像代码一样进行版本管理。
- **选择策略 (Selecting)**：只保留最相关的信息。通过“检索-重排-裁剪”的流程，对抗“Lost in the Middle”效应，消除冗余，确保送入上下文的都是精华。
- **压缩策略 (Compressing)**：在不丢失核心语义的前提下缩短上下文。可将长文档生成为面向任务的摘要，或在LLM处理前先行压缩上下文。
- **隔离策略 (Isolating)**：将不同类型的上下文信息分离管理。例如，将系统指令、知识库、工具调用结果等分层或分通道管理，避免相互干扰。

### 📋 生产环境部署的六大关键技术

将上述策略落地，以下六大技术是让系统在生产环境中稳定输出的关键：

1.  **选择性检索 (Selective Retrieval)**：通过相关性重排（如交叉编码器）、冗余消除和任务感知过滤（利用元数据），只把高价值信息送入上下文。
2.  **上下文压缩 (Context Compression)**：对检索到的长文档进行摘要或提取关键信息，确保上下文窗口中的每个token都物有所值。
3.  **层次化布局 (Hierarchical Layout)**：在上下文中合理安排信息的呈现顺序和结构，帮助模型区分信息的优先级，例如将最重要的系统指令放在开头。
4.  **动态查询重构 (Dynamic Query Reformulation)**：在将用户问题送入检索系统前，先由模型对其进行改写和澄清，以提高检索的准确性。
5.  **记忆注入 (Memory Injection)**：跨会话注入用户的长期偏好和历史状态，让AI具备“记住”的能力。
6.  **工具感知 (Tool Awareness)**：让模型在进行推理时，能实时感知和调用外部工具及数据。

### 🏗️ 一个有效的上下文分层架构

在实践中，一个有效的上下文分层架构通常由以下四层构成：

- **静态层 (Static)**：品牌调性、写作规范等固定不变的硬编码信息。
- **动态层 (Dynamic)**：当前用户、项目等会话或用户专属的实时数据。
- **记忆层 (History/Memory)**：之前的交互历史、决策和指令。
- **实时信号层 (Real-Time Signals)**：来自API或实时数据源的最新信息。

### 🛠️ 主流工具与平台

- **LangChain & LangGraph**：提供了实现上下文工程的核心工具，如`PromptTemplate`模块、`StateGraph`状态机等。
- **LangSmith**：用于追踪、评估和版本管理上下文。
- **Redis**：提供语义缓存（如Redis LangCache），可降低高达**73%** 的推理成本。
- **AWS Bedrock AgentCore**：提供从上下文检索到管理的完整框架。
- **MCP (Model Context Protocol)**：标准化AI与外部工具、数据源的连接。

### 💎 企业级实践的关键要点

- **建立以“整个运行过程”为单位的成本意识**：Agent的成本不能只看单次对话，而要看整个任务流程。因为每一轮对话都可能重新发送完整的历史记录，导致成本呈指数级增长。一个有效的对策是使用**语义缓存**来避免重复计算。
- **在工具输出进入上下文前进行过滤**：工具调用的返回结果通常是上下文膨胀的主要原因。应在信息进入上下文窗口前进行预处理和裁剪，避免其撑爆上下文。
- **将提示词视为需要测试和版本管理的“一等公民”**：用对待代码的态度来管理提示词，为其建立测试、评估和版本管理流程。

上下文工程的演进也催生了更前沿的范式，例如**Harness Engineering（驾驭工程）**，它更进一步，通过构建完整的运行环境来约束和引导AI。以及**Meta Context Engineering（元上下文工程）**，它让AI学会自己优化上下文策略。

# 最佳实践

Harness Engineering（驾驭工程）的最佳工业实践，其核心是从“**优化模型**”转向“**设计系统**”。它的目标是为AI智能体（Agent）搭建一个集**约束、验证、反馈**于一体的工程化环境，将模型的强大但不确定的智能，转化为可靠、可控的生产力。

下面是从多个真实项目中提炼出的核心实践原则与架构方法。

### 🧭 核心指导原则

这些原则是构建稳定Harness的基石：

1.  **将代码仓库视为“真相的唯一来源”**：仓库应包含Agent工作所需的一切（如架构决策、API规范、测试用例、`AGENTS.md`指引文件）。不在仓库里的东西，对Agent来说就不存在，这确保了所有Agent基于一致的信息行动。
2.  **状态落盘，而非驻留内存**：为防止长时间运行中的上下文断裂和“失忆”，必须将Agent的中间状态、决策和执行结果持久化到外部数据库或文件系统。这样Agent就能中断恢复，保证长任务的一致性。
3.  **生成与评估分离**：不要让同一个Agent既当运动员又当裁判员。将“执行工作的Agent”与“评判工作的Agent”分离，引入专门的审查员（Reviewer）角色，通过多智能体的“张力”来保障质量。
4.  **机械强制优于文本规劝**：依赖代码（如脚本、检查器、CI/CD）而非自然语言来强制规则。例如，强制要求Agent在完成任务前必须运行编译和测试脚本，这种“硬约束”比在提示词中“软劝导”可靠得多。

### 🏗️ 参考架构：七层模型

一个成熟的Harness实践通常遵循一个分层的架构模型：

- **项目搭建层**：提供标准化模板仓库、环境隔离（如容器化）和自动化初始化脚本。
- **上下文工程层**：构建知识图谱，实现动态上下文注入和版本控制，确保模型能获取精准信息。
- **约束与防护层**：建立输出过滤、权限控制（RBAC）、熔断机制和**安全沙箱**。其核心是通过“Rule”（规则）设定工程准则，并配合脚本进行强制校验。
- **多智能体架构层**：定义标准化的角色（如规划者、生成者、审查者）和通信协议。
- **评估与反馈层**：建立包含自动化测试、质量门禁的闭环系统，持续优化Agent表现。
- **长时间任务层**：通过状态管理和断点续传技术，支持复杂业务流程的完整执行。
- **可观测性层**：建立全面的日志、指标和追踪体系，让Agent的决策过程透明、可审计。

### 🚀 实施路线图

建议采用**渐进式**的策略来落地Harness Engineering：

1.  **Level 0：实验阶段**：适用于一次性脚本或原型验证，不强调可维护性。
2.  **Level 1：建立约束**：为个人项目或MVP设定基本的“Rule”（规则），通过脚本强制验证（如编译通过、测试通过）。
3.  **Level 2：构建反馈**：适合初创团队。引入CI/CD、自动化测试和代码检查工具，形成“提交代码-自动检查-反馈结果”的闭环。
4.  **Level 3：专业分工**：针对复杂系统，将AI拆分为规划者、编码者、测试者等专业角色，管理不同上下文，避免回归错误。
5.  **Level 4：系统自治**：适用于平台级应用，系统能在无人干预下持续自我修复和重构。

### 🧩 关键实现要素

- **`Rule`（规则）**：为AI设定的“工程准则”，是“软约束”。需注意Rule可能被遗忘或绕过，因此需要用脚本等“硬关卡”来兜底。
- **`Skill`（技能）**：封装特定任务流程的“能力包”，让Agent能按标准作业程序工作。
- **`Sub-Agent`（子智能体）**：负责专业分工，通过多智能体协作分解复杂任务。
- **`Workflow`（工作流）**：定义Agent执行任务的流程和顺序。
- **`MCP` (模型上下文协议)**：标准化AI与外部工具、数据源的连接方式。
- **`Script`（脚本）**：用于实现自动化校验的“硬关卡”。

### 💎 总结

Harness Engineering的最佳工业实践，本质上是将AI应用开发从一门“艺术”转化为一门可重复的“工程”。它要求开发者从“写提示词的人”转变为“设计系统、制定规则、把控质量的人”。

建议从最基础的**约束与防护层**开始，逐步引入**多智能体架构**和**评估反馈闭环**，最终目标是让你的AI系统变得可靠、可预测，并能持续产出高质量的结果。
