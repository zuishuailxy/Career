# 理解 Token、上下文窗口、输入输出成本

你发给模型的内容会先被切成 Token，模型能一次“看到”的 Token 总量上限就是上下文窗口，而计费通常就是按输入 Token 和输出 Token 分开计算。

## 什么是token

模型处理文本时的最小切分单位。

英语： 1 Token ≈ 0.75 个单词
中文 1000 汉字 ≈ 1200~1800 Token

### GPT 为什么要拆 Token？

因为神经网络处理的是数字

Token
↓
Embedding
↓
Vector

## 什么是上下文窗口（Context Window）

上下文窗口就是模型一次请求里最多能处理的 Token 容量。

它通常包括：

- system prompt
- user prompt
- 历史对话
- 检索到的知识片段
- tools/function schema
- 最终模型输出

窗口不是只给用户问题用的，而是所有东西一起共享。

窗口可以理解成缓存区，超过窗口，最高的内容会被挤出去，类似先进先出队列(FIFO)


为什么长对话会失忆？

超出窗口,最前面的内容被裁剪,所以GPT 忘记了

## 输入 Token 和输出 Token

大多数模型的 API 计费，会分成两部分：
- 输入成本：你发给模型的 Token
- 输出成本：模型返回给你的 Token

大多数模型： Output > Input 输出更贵

总成本=输入Token数×输入单价+输出Token数×输出单价

这也是为什么实际做产品时，成本控制重点不只是“少调用”，而是：

- 减少无效上下文
- 控制历史消息长度
- 控制 RAG 召回数量
- 控制输出长度
- 区分贵模型和便宜模型的使用场景


## 面试级理解

如果面试官问：

Token、Context Window、成本之间有什么关系？

可以回答：

GPT 以 Token 为基本处理单位。上下文窗口决定模型一次能够看到的 Token 总量，包括系统提示、历史消息、检索内容和模型输出。API 成本通常按输入 Token 和输出 Token 分别计费，因此上下文越大、输出越长，成本越高。在 Agent 系统中，需要通过上下文压缩、历史摘要和 RAG 检索等方式控制 Token 消耗，提高性能和降低成本。
