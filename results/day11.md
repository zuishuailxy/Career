# Conversation Memory

在大语言模型（LLM）中，**对话记忆（Conversation Memory）** 指的是模型在连续多轮对话中，能够记住、理解并引用之前交流过的信息，从而生成连贯、上下文相关且有逻辑的回复的能力。

与人类对话不同，大多数大模型本身是**无状态**的——它们不会天然地“记得”你上一句说了什么。每一次调用 API 或发送请求，模型都只看到当前输入。因此，实现对话记忆的核心在于由**开发者/应用层主动管理对话历史**，并将历史信息作为模型输入的一部分回传。

## 什么是 Conversation Memory？

Conversation Memory = 让模型记住当前会话中已经发生过的内容。

## GPT 真的会记忆吗？

不会，当我们把聊天历史记录一起发给大模型时，大模型根据聊天历史，来预测下一个 Token。

因此：

> 所谓 Conversation Memory，本质是把历史消息重新发给模型。

## 为什么会遗忘

大模型之所以会“遗忘”，本质上并非像人类那样大脑衰退，而是源于其**技术架构与使用方式的固有特性**。简单说：模型本身**从不主动记住**，它的每一次“回忆”都依赖于我们提供的输入。

以下是导致“遗忘”的核心原因：

### 1. 模型天生无状态 —— 根本原因

大模型本身就像一个**极其聪明但患有“瞬时失忆症”的专家**。它处理完你当前的问题后，不会自动保留任何关于这次对话的信息。下一次你问它时，它的“大脑”里只有你刚发来的那一条消息，之前的对话早已消失。  
👉 **记忆全靠开发者主动提供**：每次调用都必须把历史对话重新放进输入里，模型才能“看到”过去。

### 2. 上下文窗口有限 —— 硬件天花板

即使我们每次都把完整对话历史发给模型，模型能一次性处理的信息量也是有上限的，称为**上下文窗口**（例如 4096、8192、128k 个 token）。

- 当对话轮次过多、总 token 超过窗口时，最早的消息就会被**强制截断丢弃**，模型自然就“忘记”了开头的内容。  
  👉 **窗口就像模型的一次性工作台**，它只能摆下那么多材料，多出来的只能扔掉。

### 3. 注意力机制的“资源分配” —— 潜意识忽略

即便历史信息在窗口内，模型内部的**注意力机制**也可能在生成长回复时，把注意力集中到最新的消息或关键位置，而对较早的细节“关注不足”。这会导致虽然信息存在，但模型回答时**像没看到一样**，表现出遗忘。  
👉 类似人类开会时，虽然前面说过某个数据，但讨论到后来大家还是忘了。

### 4. 训练方式导致缺乏长期记忆能力

大模型的训练主要基于**独立样本**（如一个文档、一个对话片段），而不是长时间连续的对话流。它被训练成根据“紧挨着的上文”生成下文，而不是有意识地记住跨越多轮的用户偏好、事实声明。  
👉 模型没有“我想把用户叫张三这件事存进长期记忆”的机制，它只是机械地处理当前输入。

### 5. 应用层的主动截断/总结 —— 工程取舍

为了让对话能在有限成本内持续，开发者通常会主动实现**遗忘策略**：

- **滑动窗口**：只保留最近 N 轮，丢弃旧轮。
- **总结压缩**：用模型把长历史浓缩成一段摘要，原细节被“遗忘”。  
  这些策略虽能控制 token 消耗，却也必然导致部分信息丢失。
- **截断**：从最早的消息开始丢弃。
- **向量检索记忆**：将历史嵌入向量数据库，每次检索最相关的若干条记忆，放入上下文（常用于长期记忆或知识库对话）。

---

### 结论

大模型的“遗忘”不是缺陷，而是**设计上的代价**。要实现更持久的记忆，需要通过**外部存储**（向量数据库、长期记忆模块）和**检索增强**来模拟人类的长期记忆，让模型在需要时主动“想起”相关信息。

## 如何实现对话记忆

最常用的方法是 将整个对话历史打包进模型的输入。

### 1. 维护消息列表

使用对话格式

```python
messages = [
    {"role": "system", "content": "你是一个助手。"},
    {"role": "user", "content": "我叫张三"},
    {"role": "assistant", "content": "你好张三！"},
    {"role": "user", "content": "我的名字是什么？"}
]
```

每次调用时，将这个列表发送给模型。模型基于列表中的全部消息生成下一个回复。

优点实现简单，缺点 Token 消耗越来越大

### Summary Memory

定期总结 让模型定期总结并提炼用户偏好、事实、目标，替换原始历史

例如：

历史：

```
用户：
学习 FastAPI

用户：
学习 RAG

用户：
学习 Agent
```

---

压缩成

```
用户正在学习 AI 开发：
- FastAPI
- RAG
- Agent
```

从而减少 Token

### Vector Memory（现代 Agent 常用）

把历史消息通过 embedding 转换为向量，然后存入数据库：

```
Embedding
↓
Vector DB
```

用户提问时候，系统：

```
Embedding Query
↓
相似度检索
↓
找回相关记忆
```

## Conversation Memory vs Long-term Memory

### Conversation Memory

作用范围： 当前聊天， 当前会话结束，记忆消失

### Long-term Memory

作用范围：跨会话， 当前会话结束，记忆保存

## 高级记忆技术

- **长期记忆**：利用外部数据库（如 Chroma、Pinecone）存储用户历史交互，通过向量相似度检索出最相关的对话片段注入上下文，实现跨会话的“记住”。

- **记忆反思**：让模型定期总结并提炼用户偏好、事实、目标，存入长期记忆。

- **可编辑记忆**：允许用户显式添加/删除/修改记忆内容（类似 AI 的“记忆管理面板”）。
